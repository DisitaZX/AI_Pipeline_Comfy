Выступи в роли профессионального сценариста и AI-режиссера. Я буду делать из главы ранобэ короткое вертикальное видео (Shorts/TikTok/Reels) в стиле modern anime. Озвучка через qwen3tts (китайский аниме-рассказчик), картинки — Z Image Turbo, видео — WAN 2.2.

Задача выполняется в ДВА ПРОХОДА. Первый проход — извлечение, второй — сборка. Между проходами не делай выводов, отвечай ровно как сказано.

================================================================
ПРОХОД 1 — ИЗВЛЕЧЕНИЕ КЛЮЧЕВЫХ ЭЛЕМЕНТОВ ИЗ ГЛАВЫ
================================================================

Прочитай текст главы (он в самом конце этого промпта) и заполни JSON следующего вида. Эти элементы ОБЯЗАНЫ потом попасть в озвучку и в раскадровку — это «контракт верности первоисточнику».

{{
  "key_elements": {{
    "characters": [
      {{"name": "имя из главы", "role": "роль в этой главе одной фразой", "must_show": "что обязательно про него видно в видео"}}
    ],
    "settings": [
      {{"place": "локация из главы", "details": "что в ней особенного (модерн/древнее, освещение, ключевые объекты)"}}
    ],
    "numbers_and_stats": [
      "все важные числа, ранги, цены, сроки — слово в слово как в главе"
    ],
    "items_or_objects": [
      "все названные артефакты/предметы/техники с краткой ремаркой"
    ],
    "direct_quotes": [
      "минимум 4 прямые реплики или ключевые фразы из главы verbatim — это якоря для голоса автора"
    ],
    "plot_beats": [
      "1 строка на каждую сюжетную точку, в порядке главы"
    ],
    "visual_hooks": [
      "что зрелищно для аниме-формата — голограммы, золотые столбы света, хищные ухмылки, и т.п."
    ],
    "stakes_and_motivation": "одной фразой: что движет ГГ и что он рискует потерять"
  }}
}}

Правила Прохода 1:
- Не сокращай. Если в главе названы 6 предметов из магазина — выпиши все 6.
- Не интерпретируй. Если в главе сказано «8888», пиши «8888», не «огромная сумма».
- direct_quotes копируй ТОЧНО (в т.ч. знаки препинания и системные скобки [...]).
- В plot_beats не пропускай ни одной сюжетной точки, даже если она кажется «технической» (открытие магазина, проверка баланса, покупка пилюли — это всё точки).

================================================================
ПРОХОД 2 — СБОРКА ОЗВУЧКИ И РАСКАДРОВКИ
================================================================

Используя ТОЛЬКО `key_elements` из Прохода 1 как обязательный контракт, собери финальный JSON со всем сценарием.

ШАГ 2.1: Текст для озвучки (`tts_text`)
Хронометраж: 320–400 слов на русском (это ~2:30–2:55 живой речи рассказчика).
Жёсткие правила:
- В tts_text должны попасть ВСЕ элементы из `numbers_and_stats`. Никакие числа не теряются.
- В tts_text должны попасть минимум 4 объекта из `items_or_objects` с ценой/характеристикой.
- Минимум 3 фразы из `direct_quotes` встроены verbatim (можно с минимальной адаптацией под устную речь, но смысл и ключевые слова сохранены).
- Все `plot_beats` должны прозвучать в правильном порядке. Не пропускай ни одной сюжетной точки.
- Стиль — кинематографичный, эпичный, от первого лица если в главе так, иначе от третьего. Не превращай в чисто внутренний монолог если в главе была сцена с действиями и предметами — описывай и предметы, и действия.
- Запрет на «обобщения вместо специфики». Плохо: «у него были крутые товары». Хорошо: «Демоническое Искусство Поглощения Небес — пятьдесят тысяч очков».

ШАГ 2.2: Базовые переменные `base_prompts`
Создай переменные для всех уникальных персонажей, локаций и ключевых объектов из `key_elements`. Каждая переменная — 1 абзац подробного визуального описания на английском с маркерами modern anime style. Эти переменные подставляются в скобках, например `[main_character]`. Имена переменных — `snake_case`, без пробелов. Если переменная это какое-то окружение, то не надо в его описание засовывать персонажей.

ШАГ 2.3: Раскадровка `scenes`
Жёсткие правила:
- Количество сцен: НЕ меньше 20, НЕ больше 30.
- НЕ режь предложения на отдельные сцены. Минимальная длина `text_scene` — 9 слов. Но максимальная - 30 слов.
- Каждая сцена покрывает один cohesive момент: одно действие, одна реакция, один объект в фокусе. Не клей в сцену два разных кадра.
- Распределение `text_scene` ВДОЛЬ `tts_text` — без пропусков и без дублей. Сложенные `text_scene` в порядке `scene_id` должны давать tts_text слово в слово.
- `image_prompt` (английский, modern anime style): композиция, освещение, статичная поза, выражение лица, ключевые объекты в кадре. Используй переменные в `[квадратных скобках]`. Не перегружай: video-комикс, не более 2–3 главных объектов в кадре.
- `video_prompt` (английский): ТОЛЬКО движение, поведение камеры, эффекты. Не повторяй внешность. 1–2 предложения.
- Дополнительно `previous_recap` — 1 короткое предложение по-русски, что произошло в предыдущих сценах. Это контекст для генерации видео-промптов в моём пайплайне. Для scene_id=1 поставь пустую строку.
- Дополнительно `must_carry_over` — массив из `key_elements` (числа, объекты, реплики), которые ОБЯЗАТЕЛЬНО присутствуют в этой сцене визуально или в озвучке. Пустой массив если сцена чисто переходная.
- Дополнительно `entities` — список `base_name` из `base_prompts`, которые присутствуют в этой сцене (используются в `image_prompt`). Этот список нужен для генерации картинки: каждой entity соответствует канон-референс, который будет передан в multi-image edit-модель. Жёсткие правила:
  - Каждый `base_name`, упомянутый в `image_prompt` через `[...]`, должен быть в `entities`. И наоборот: каждый элемент `entities` должен быть упомянут в `image_prompt` хотя бы раз.
  - Максимум 3 entity на сцену (hard limit edit-модели). Если в сцене визуально нужно больше — упрости композицию или раздели на две сцены.
  - Имена в `entities` должны точно совпадать с `base_name` из `base_prompts` (snake_case, без скобок).
  - Просто так переменную добавлять в image промпт нельзя. Например: есть base prompt: character, в сцене присутствует 4 объекта включая его, мы берём 3 объекта исключая из их числа character, так как есть ограничение на 3 entity на сцену. В этом случае ни должно быть никакого упоминания character в сцене.
  - в video_prompt вообще нельзя вставлять переменные ни в каком виде, если необходимо описать, что происходит используй общие термины типа - man on the right side, girl on the middle.

================================================================
ФИНАЛЬНЫЙ ОТВЕТ
================================================================

Ответь СТРОГО одним JSON-объектом без markdown и пояснений. Структура:

{{
  "tts_text": "...",
  "base_prompts": [
    {{"base_id": 1, "base_name": "main_character", "base_prompt": "..."}}
  ],
  "scenes": [
    {{
      "scene_id": 1,
      "previous_recap": "",
      "must_carry_over": ["8888 очков Удачи", "Е Чэнь"],
      "entities": ["main_character", "shop_interior"],
      "text_scene": "...",
      "image_prompt": "...",
      "video_prompt": "..."
    }}
  ]
}}

САМОПРОВЕРКА перед отправкой ответа:
1. Все числа из `key_elements.numbers_and_stats` встречаются в `tts_text`? Если нет — переделай tts_text.
2. Минимум 4 предмета из `key_elements.items_or_objects` названы в `tts_text` с ценой/характеристикой? Если нет — переделай.
3. Минимум 3 цитаты из `key_elements.direct_quotes` присутствуют в `tts_text` (можно адаптированно)? Если нет — переделай.
4. Сложенные `text_scene` дают исходный `tts_text` без потерь? Если нет — пересегментируй.
5. У каждой сцены `text_scene` >= 12 слов? Если нет — объедини с соседней.
6. `previous_recap` сцены N описывает события сцен 1..N-1? Если нет — допиши.
7. У каждой сцены есть `entities`? Если нет — добавь.
8. `len(entities) <= 3` для каждой сцены? Если больше — упрости композицию или разбей сцену.
9. Каждый `[base_name]` в `image_prompt` сцены присутствует в её `entities`, и наоборот — каждый `entities[i]` упомянут в `image_prompt` через `[base_name]`? Если нет — исправь несоответствие.
10. Все имена в `entities` точно совпадают с `base_name` из `base_prompts`? Если есть опечатки или неизвестные имена — исправь.

Только после прохождения самопроверки выводи JSON. Никакого текста вне JSON.
